# MEMO-Appflow 调研报告

---

## 一、研究背景

随着移动设备承载的工作负载日益复杂，用户在不同应用之间的切换频率显著增加。应用冷启动延迟和系统资源利用率低下成为影响用户体验的突出问题。传统的被动式资源管理依赖操作系统在内存压力触发后才进行回收，缺乏对用户行为模式的主动感知，导致频繁的冷启动与不必要的内存换入换出。

预测引导的资源管理（Prediction-Guided Resource Management）旨在通过建模用户应用使用习惯，提前预测下一个可能被打开的应用，并据此主动调度系统资源（预加载、保活、回收），从而在用户无感知的情况下缩短启动延迟、提升系统流畅度。为进一步提升预测精度，系统可融合应用切换时序、时间规律、系统状态、用户交互模式与场景上下文等多模态语义信息，构建对用户意图的深度理解框架。该方向涉及操作系统资源管理机制、时序行为建模、多模态语义融合以及策略决策等多个层面，具有较强的系统研究价值。

## 二、国内外研究现状

### 2.1 AppFlow：预测引导的 Android 资源调度

AppFlow 是当前该领域最具代表性的系统级工作之一。其核心贡献不仅在于应用预测，更在于将预测结果转化为具体的系统资源决策，并深入 Android Framework 和 Linux Kernel 实现对应的调度机制。AppFlow 提出三大协调模块：

- **选择性文件预加载器（Selective File Preloader）**：基于预测结果，在应用启动前按预算预加载热点小文件及大文件分段，降低冷启动 I/O 延迟。
- **自适应内存回收器（Adaptive Memory Reclaimer）**：根据内存压力和分配速率动态调整回收策略，在预测窗口期内保护预加载集合。
- **上下文感知进程杀死器（Context-Aware Process Killer）**：结合应用使用频度、内存膨胀量和重新启动基线，对后台进程进行智能排序与延迟杀死，减少无效保活带来的内存浪费。

AppFlow 的关键参数包括：30 分钟近期使用窗口、100 MB 预加载预算、100 ms 分配活动采样周期。这些参数为后续复现提供了明确的实验基准。

### 2.2 ATPP：基于深度标记时序点过程的移动应用预测

ATPP（App Prediction System Based on Deep Marked Temporal Point Processes）将应用使用建模为标记时序点过程（Marked Temporal Point Process），同时预测下一个应用的身份和打开时间。该工作表明，引入时间感知可以显著提升预测质量，对预加载和保活的时机选择具有直接指导意义。ATPP 为我们扩展当前 Markov 基线向时序感知模型演进提供了理论参考。

### 2.3 数据集与评测基准

公开数据集方面，**LSApp**（Large dataset of Sequential mobile App usage）提供了 599,635 条应用使用记录、76,247 个会话、292 名用户、87 款应用，覆盖平均 15 天的使用周期。该数据集包含用户 ID、会话 ID、时间戳、应用名称和事件类型（打开、关闭、交互、损坏），适合用于离线模型训练与回放评测。LSApp 使用应用名称而非 Android 包名，适合作为离线基准，与在线设备模式形成互补。

## 三、关键技术调研

### 3.1 Android 系统资源管理机制

Android 系统的应用生命周期由 ActivityManagerService 与 Low Memory Killer（LMK）协同管理。后台应用根据内存压力被逐级回收，而应用启动则涉及 Zygote 进程分叉、类加载、资源解压与页面映射等多个阶段。冷启动延迟的主要来源包括：

- Dex 文件加载与 JIT/AOT 编译
- 资源文件（图片、布局、字符串）解析
- SharedPreferences 与数据库初始化
- 网络请求与缓存预热

在普通应用沙箱内，直接干预进程优先级、内存页回收或文件预读需要特权；但通过 `UsageStatsManager` 采集应用启动历史、通过 `ActivityManager.getMemoryInfo()` 获取内存快照、通过 `Intent` 启动目标应用均属于公开 API，可行性高。

### 3.2 系统级行为采集技术

调研了两种主要的行为采集路径：

- **应用层采集**：Android `UsageStats` API 提供系统级应用启动历史，无需 root，数据粒度为包名级别，足以支撑应用切换序列建模。
- **进程级采集**：Linux eBPF、macOS DTrace、Windows ETW 可捕获 `fork`/`exec`/`exit` 等进程生命周期事件。该路径信号更丰富，但受平台差异和权限限制影响较大，在 Android 端需要 root 或自定义 ROM 支持。

综合评估后，确定以 `UsageStats` 作为在线采集主路径，以进程级追踪作为未来系统级扩展的备选方案。

### 3.3 预测模型技术

- **Markov 链**：基于应用转移频率建立一阶或多阶转移矩阵，实现简单、推理开销极低，适合作为基线模型和快速回退策略。
- **Transformer 时序模型**：具备长程依赖捕捉能力，在 LSApp 数据集上表现出较高的预测精度（Hit@1 可达 66% 以上）。项目已引入 C++ 实现的轻量 Transformer 推理引擎，单步延迟在 11–45 ms 之间，内存占用约 200 MB，可作为设备空闲时的主力模型。
- **Circuit Breaker 机制**：针对移动设备资源受限的特点，调研了在设备空闲时启用大模型、资源紧张时回退至 Markov 基线的动态切换策略，以平衡精度与功耗。

### 3.4 资源调度与策略引擎

策略引擎负责将预测结果映射为系统资源决策。调研了以下策略范式：

- **阈值策略（Threshold Policy）**：根据预测分数与预设阈值（keepAlive、prewarm、hint）直接输出决策列表，逻辑清晰，易于调试。
- **AppFlow 策略**：引入内存压力模式（NORMAL / EFFICIENCY_FIRST / REBALANCE）、预加载预算、应用启动画像（Launch Profile）和 Kill Candidate 排名，决策维度更丰富，更贴近系统级调度需求。
- **Budget-Aware / Energy-Aware 策略**：在资源或电量受限场景下对决策进行裁剪与降级。

## 四、数据集调研

| 数据集 | 规模 | 特点 | 适用场景 |
|-------|------|------|---------|
| LSApp | 599K 记录，292 用户，87 应用 | 真实用户研究，时序完整 | 离线训练、回放评测、基线对比 |
| 自建在线采集 | 依赖设备 UsageStats | 真实包名，可启动验证 | 在线模式、真机演示、个性化建模 |

项目同时支持 CSV 与 JSON 格式的自定义数据集导入，并内置了数据集注册表（DatasetRegistry）、分割管理器（DatasetSplitManager）和归一化流水线，确保离线数据与在线采集共享同一套 `AppEvent` 内部 schema。

---

## 五、结论

通过调研，我们确认以下技术路径在当前 Android 生态下具备可行性：

1. **应用层采集 + 公开数据集回放** 双轨并行，可覆盖在线演示与离线复现两种需求；
2. **Markov 基线 + Transformer 增量** 的预测架构，兼顾实时性与精度；
3. **Threshold 策略 + AppFlow 调度逻辑** 的策略分层，可从简单规则平滑过渡到复杂系统调度；
4. **App 级沙箱原型 → JNI/C++ seam → 未来 Binder/AOSP 集成** 的渐进式系统扩展路线，符合 Android 平台权限现实。

此外，融合多模态用户行为语义（时间规律、系统状态、交互模式、场景上下文）的预测框架已被确认为提升模型精度的有效路径，上述调研结论已直接指导项目的架构设计与代码实现。
